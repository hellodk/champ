A — Inline Diff in the Panel

Diffs render directly in the right pane of the workflow tab. Accept/Reject buttons per file. "Accept All" and "Reject All" at the bottom. No VS Code diff editor involved.

Pros

  • Never leaves the workflow view
  • Fast review — one panel, all changes
  • Accept/reject per file, per hunk

Cons

  • Custom diff renderer needed
  • Less familiar than VS Code diff

B — Opens VS Code Diff Editor per File

The panel lists changed files. Clicking any file opens VS Code's native diff editor as a new tab. Accept/Reject All buttons remain in the panel.

Pros

  • Uses VS Code's proven diff UI
  • Inline editing possible in the diff
  • No custom renderer needed

Cons

  • Leaves the workflow view
  • Multiple tabs get cluttered

C — Split Review Panel (Recommended)

Horizontal pipeline bar at top. Below: file list on left (click to select), inline diff on right with per-file Accept/Reject. Accept All / Reject All to move on.

Pros

  • Everything in one view — no tab jumping
  • File list shows review progress
  • Inline diff with per-file granularity
  • Closest to Cursor's experience

Cons

  • Custom diff renderer needed
  • Most complex to build

C is the most cohesive enterprise experience. B reuses VS Code infrastructure. A is a middle ground.
